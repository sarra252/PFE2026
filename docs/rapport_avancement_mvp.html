Rapport d'avancement - Chatbot SQL Teradata (MVP)

Date: 10 mars 2026

1. Contexte et objectif

Le projet vise la realisation d'un chatbot backend capable de generer, documenter et optimiser des requetes SQL orientees Teradata. En raison de l'absence initiale d'acces a un environnement Teradata reel, une approche simulation/mock a ete retenue pour livrer un MVP fonctionnel, testable et evolutif.

2. Realisations techniques

  • API FastAPI operationnelle avec les routes: /health, /version, /generate-sql, /document-sql, /optimize-sql.
  • Contrat de reponse standard sur toutes les routes: request_id, status, data, warnings, errors.
  • Authentification simple par API key via l'en-tete x-api-key.
  • Middleware d'observabilite avec correlation id et mesure de latence (X-Request-ID, X-Process-Time-Ms).

3. Securite SQL et logique metier

  • Validation SQL en lecture seule: seules les requetes SELECT et WITH ... SELECT sont autorisees.
  • Blocage des mots-cles dangereux: DROP, DELETE, UPDATE, INSERT, MERGE, ALTER, etc.
  • Blocage des requetes multi-statements.
  • Warning automatique sur l'utilisation de SELECT *.
  • Service de generation SQL (mock LLM) retournant SQL + explication + score de confiance.
  • Service de documentation SQL: extraction tables/colonnes et logique des clauses.
  • Service d'optimisation SQL: heuristiques initiales et recommandations.

4. Tests et validation

  • Fichier de tests automatise tests/test_api.py avec 10 tests.
  • Couverture: disponibilite API, succes endpoints, auth, blocage SQL dangereux, blocage multi-statements, format de reponse, headers de tracabilite, non-regression sur 10 prompts de reference.
  • Resultat obtenu: 10 passed.
  • Validation manuelle Postman realisee sur 5 scenarios (generation, documentation, optimisation, erreur auth 401, erreur securite SQL 400).

5. Livrables produits

  • Documents projet: charte, backlog, architecture, runbook demo, guide Postman, next steps.
  • Prompts: generate_sql_v1.md et generate_sql_v2.md.
  • Artefacts demo: collection et environnement Postman.

6. Incidents rencontres et resolution

  • Erreur d'encodage Unicode corrigee dans les fichiers Python concernes.
  • Dependance de test manquante (httpx) installee pour le TestClient.
  • Verification finale confirmee par execution des tests en local.

7. Pourquoi cette approche

La strategie suivie est incrementalement orientee reduction de risque: d'abord un socle API stable, ensuite la securite SQL, puis la validation par tests et scenarios Postman, enfin la preparation de l'integration Teradata reelle. Cette demarche garantit un MVP demonstrable rapidement sans compromettre la qualite technique.

8. Etat actuel et prochaine phase

Le MVP est operationnel, teste et pret pour la phase 2. Les prochaines actions sont: connexion Teradata reelle (teradatasql/ODBC), branchement d'un vrai provider LLM, ajout d'une evaluation de plans d'execution (EXPLAIN) si disponible, et extension des tests d'integration/performance.
